# Plan: Flowchart/Pseudocode вҶ” Java Alignment Verifier (Kaggle T4Г—2)

_Snapshot: 2026-07-17 PM. Mirrors the working plan; the CURRENT INCREMENT section is
what is being implemented now._

## Context

BUU / SuperAI Season 6 hackathon (individual, submissions on Kaggle, closes Sun 19 Jul 2569).
Build a system that scores **algorithmic consistency (0вҖ“3)** between a program's design
artifact and its Java source, for two sub-tasks:

- `flowchart.png` вҶ” `solution.java`
- `pseudocode.txt` вҶ” `solution.java`

**Rubric (from the brief, page 8 вҖ” behavioral, not syntactic):**
- **0 Inconsistent** вҖ” clearly different, or a different solution approach.
- **1 Weakly consistent** вҖ” some parts match but several important differences (steps, conditions).
- **2 Mostly consistent** вҖ” main approach matches; minor differences or missing detail.
- **3 Fully consistent** вҖ” concept, steps, conditions, and execution order all match.
- Variable names / writing style need **not** be identical if behavior is equivalent (page 7).

**Hard constraints that shape every decision:**
- **No commercial APIs** (no OpenAI/Claude/Gemini). Open-source models only.
- Current setup runs **internet-ON**: model downloads from HF hub via the `git clone` Kaggle
  workflow. `ALIGN_OFFLINE=1` switches to offline/local-weights mode.
- **T4 (Turing) hardware**: **no bfloat16, no FlashAttention-2** вҶ’ load `float16` +
  `attn_implementation="sdpa"`. 16 GB/GPU.
- Must not modify input artifacts; must not hard-code test answers.
- **Metric = exact-match accuracy** вҶ’ be decisive (no hedging to the middle); majority-vote
  self-consistency and prompt calibration directly buy accuracy.

**Key data reality:** training labels are tiny (33 cases). This is **too little to train a
classifier or reliably LoRA-tune** вҶ’ the solution is **LLM-as-judge with a strong rubric
prompt**, using labeled cases as few-shot anchors + a validation set. LightGBM / heavy
fine-tuning are deferred (step 5, optional).

**Model decision:** Primary judge = **Qwen2.5-VL-7B-Instruct** (one model handles image+Java
and text+Java). Note: "Qwen3.5-4B" does not exist; the real alternatives are Qwen2.5-VL-3B/7B or
Qwen3-VL-4B/8B. Sticking with 2.5-VL-7B for T4 reliability.

## CURRENT INCREMENT вҖ” Baseline B (Mermaid two-stage) + Ensemble

**Why:** Direct image judging (Baseline A) sometimes misreads flowchart *structure* вҖ” the
thing the graders actually score (e.g. case_01: combined `a>=b AND a>=c` conditions in the
design vs nested pairwise in the code вҶ’ human score 0). A two-stage path gives an independent,
structure-first vote: the VLM first converts the flowchart to **Mermaid** (explicit
nodes / edges / Yes-No branches), then the rubric judge compares that graph against the Java.
Ensembling **A + B + the rule-based structural check** by majority vote smooths single-method
errors. (The user's sketch shows Python + MATCH/MISMATCH; this task's code is **Java** and the
output stays a **0вҖ“3 score**, which subsumes match/mismatch.)

```
                        в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ      в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ

**Flow per flowchart case (balanced budget):**
1. Preprocess PNG + OCR (existing `preprocess_flowchart`, `ocr_flowchart`).
2. **Perception (once, greedy, cached):** `image вҶ’ Mermaid` via new `flowchart_to_mermaid_turn`.
3. **Baseline A votes:** direct image judge вҖ” greedy + `SAMPLES_PER_BASELINE` (=2).
4. **Baseline B votes:** rubric judge over the Mermaid text вҖ” greedy + 2.
5. **Structural vote:** `fallback_score_from_signals(java, ocr_or_pseudo)` Г—1.
6. **Score = majority(A вҲӘ B вҲӘ structural)**; tie breaks toward Baseline A's greedy vote.

Pseudocode cases have no image, so A and B collapse вҶ’ one text-judge (greedy + 2) + structural
(no wasted Mermaid call).

## STEP 1 вҖ” Prompt Engineering

**Design: rubric-grounded chain-of-check.** One system prompt (rubric + structural-grading
principles + Thai guide + output contract), two user templates (flowchart carries the image;
pseudocode/Mermaid carries text), same 6-dimension comparison core.

**System prompt encodes:**
- Role: strict expert grader of design-vs-code algorithmic consistency.
- The exact 0вҖ“3 rubric, worded so 0 covers "same output via a different STRUCTURE."
- Grading principles: compare step-by-step / decision structure, not just "both solve it."
- Thai vocabulary map + Yes/NoвҶ’condition mapping.
- Output contract: **reason first in plain text, then JSON on the last line**.

**User prompt = chain-of-check:** summarize Java вҶ’ summarize design вҶ’ compare 6 axes
(input, output, order, loop, condition, computation, each match|partial|mismatch) вҶ’ list
mismatches вҶ’ `final_score` 0вҖ“3.

**Few-shot anchors:** from `alignment_score_training.csv`, one labeled case per score level
(0/1/2/3), with **score-consistent** demonstration answers (case_01 is the canonical 0).

**Decoding:** greedy primary pass + sampled self-consistency votes; coerce `final_score` into
{0,1,2,3}; one retry on parse failure, then rule-based fallback.

**Rule-based structural signals** (`parser/java_signals.py`): counts of inputs/loops/branches/
outputs/arithmetic injected as a "Detected in Java: вҖҰ" hint, and used for the fallback score.

## STEP 2 вҖ” Image Preprocessing

All in-memory (never overwrite the PNG). `feature/image_prep.py`:
flatten RGBAвҶ’RGB вҶ’ auto-crop + pad вҶ’ upscale small charts (Lanczos) вҶ’ mild non-destructive
contrast (NOT binarized вҖ” kills arrowheads) вҶ’ cap resolution (bound VLM tokens). Optional
**OCR augmentation** (Tesseract `eng+tha` / PaddleOCR) with `DEBUG_OCR` line printing.

## Steps 3вҖ“5 (post-baseline)

- **Step 3 В· Self-consistency:** sampled votes at `tempвүҲ0.7`, majority vote; tie вҶ’ greedy.
- **Step 4 В· Ensemble:** specified in CURRENT INCREMENT вҖ” A (image) + B (Mermaid two-stage) +

- **Step 5 В· LoRA (optional, last):** QLoRA on Qwen2.5-VL; high overfit risk with ~dozens of
  labels вҖ” attempt only if validation accuracy stalls.
